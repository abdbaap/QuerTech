Technical Mechanism: Use-After-Free Explained

  1. Chrome allocates memory for a "Font feature" object, and a "pointer" references this object in RAM.
  2. When the "Font feature" is no longer needed, Chrome frees the associated memory ("OBJECT" block in RAM).
  3. The bug occurs because Chrome does not invalidate the pointer, leaving it "dangling," still pointing to a memory location that is now marked "empty" and "available for use" by the operating system.
  4. An attacker can then make a new memory request (e.g., via malicious JavaScript), which the OS is highly likely to fulfill by reusing the recently freed block.
  5. This effectively allows the attacker to inject their malicious JavaScript code into the memory location that the original, dangling pointer still references, leading to code execution when Chrome attempts to use the old pointer.

Scope Exploitation Scope

The initial exploit runs code inside the browser tab's sandbox. Google's "exists in the wild" statement suggests attackers have a secondary exploit chain to bypass this sandbox isolation, enabling system-wide access.

A public Proof of Concept (poc.html) merely crashes the tab with "Error Code 11".

Root Cause Underlying Cause and Mitigation Trend

This vulnerability stems from C/C++'s manual memory management. Languages like Rust inherently address these memory safety issues through their ownership and borrowing model, preventing classes of vulnerabilities like UAF.

Firefox, for instance, has already implemented significant parts of its CSS parsing engine in Rust to enhance security, showcasing an industry trend towards safer languages for critical components.
